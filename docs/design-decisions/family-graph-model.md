# Design Decision: Relationship-Centric Family Graph Model

## Status

Accepted and implemented.

---

## Problem

A genealogy application must model more than a single mother/father tuple per person.

Real datasets often include:

- multiple partner relationships over time
- patchwork families
- adoption and foster relationships
- children linked to a specific relationship rather than to one timeless household

A simple `mother_id` / `father_id` model would be too rigid for these cases.

---

## Decision

ReRooted models family structure using:

1. a `relationships` table describing the adult relationship (`person1`, `person2`, type, dates)
2. a `relationship_children` association table linking one or more children to that relationship

Tree rendering is then generated by projecting these normalized records into a React Flow graph via `tree_service.build_tree()`.

---

## Why this was chosen

### Supports patchwork families

A child belongs to a **relationship instance**, not just to a flat parent pair. This makes it possible to express:

- children from different partnerships
- re-partnering after separation
- partial or uncertain family information

### Supports relationship semantics

The system preserves the distinction between:

- `partner`
- `ex`
- `adoption`
- `foster`
- `unknown`

These values affect graph output and can be surfaced in the UI.

### Maps cleanly to graph visualization

The backend’s `/tree` response can produce:

- partner edges between adults
- child edges from each parent to each child
- dashed child edges for adoption/foster relationships

This is more expressive than deriving a graph from fixed parent columns.

---

## Consequences

### Positive

- more realistic genealogy modeling
- better support for non-traditional family structures
- stable bridge to React Flow via explicit graph projection
- easy expansion for future relationship metadata

### Trade-offs

- queries are more complex than direct parent columns
- child membership requires validation to avoid duplicates/conflicts
- the projection layer must assemble the tree from multiple tables

The current service layer absorbs this complexity so the API surface stays clean.

---

## Implementation Notes

The main enforcement rules currently live in `relationship_service.py`:

- parent references must point to existing persons
- a person cannot be both parent and child in the same relationship
- duplicate child links are rejected
- the same relationship record can emit two parent-to-child edges in `/tree`

This is the core domain model behind the graph engine.
